# querycreator

## 한 줄

자연어 질문을 안전한 Oracle SELECT/저장 함수 호출로 바꿔주는 사내 셀프서비스 도구. DB 담당자에게 쏟아지던 데이터 확인 문의를 LLM 에이전트로 분산하는 게 목적이다. MCP 호환 도구 서버로 만들어 사내 LLM 게이트웨이와 에이전트 플랫폼 양쪽에서 그대로 끼워 쓸 수 있게 했다.

## 컨텍스트·동기

제조 공정 데이터를 다루는 Oracle 레거시 위에서 일한다. 시스템 담당자에게 "이 주문 진행 상태 좀 봐줘", "어제 생산량 합계", "이 공정 코드가 무슨 뜻이냐" 같은 데이터 확인 문의가 매일 들어온다. 화면을 다 만들어주는 건 비현실적이고, 즉석 쿼리/MCP를 써줘도 다음 요청이 또 온다. 근본 처방은 사용자가 직접 묻고 LLM이 알아서 쿼리 짜서 답하게 하는 것.

설계는 2026-04-09에 잡았고 구현은 2026-04-10 새벽에 첫 커밋까지 돌렸다. 04-16에 회고 세션을 한 번 더 돌리면서 사내 에이전트 플랫폼과 사내 LLM 게이트웨이 양쪽에 붙일 수 있게 도구 인터페이스를 MCP 호환으로 정리했다.

## 핵심 결정

### SELECT 전용, DML/DDL 원천 차단

대안은 (a) 운영자 승인 워크플로우로 INSERT/UPDATE도 허용, (b) 읽기 전용 DB 계정만 주고 SQL은 자유롭게. 골라낸 건 (c) 읽기 전용 계정 + 검증 레이어에서 SELECT 외 구문 거부 + 다중 구문 금지 7가지 규칙. 이유는 LLM이 손에 쥔 쿼리가 어느 순간 파괴적으로 변할 가능성을 코드 레벨에서 끊고 싶었기 때문. 규칙 1·2·7(SELECT 전용/SELECT * 금지/다중 구문 금지)은 환경변수로도 끌 수 없게 하드코딩했다.

### MCP 호환 도구 인터페이스

에이전트 플랫폼 종속 도구로 짜면 사내 LLM 게이트웨이·에이전트 플랫폼·향후 다른 LLM 클라이언트에 일일이 어댑터를 새로 만들어야 한다. 도구 시그니처를 MCP 표준에 맞춘 세 개(`get_metadata`, `execute_query`, `call_function`)로 좁히고, 클라이언트는 MCP 서버 하나만 가리키게 했다. 사내 LLM이 Claude Code 소넷·GPT-5.4·Gemini 3.1로 넘나들어도 도구 쪽은 안 건드린다.

### 메타데이터 4계층 주입 (업무사전·물리·공통코드·지식힌트)

LLM에게 스키마 전체를 그대로 던지면 토큰도 폭발하고 업무 의미도 모른다. 대안은 (a) RAG로 임베딩, (b) 운영자 큐레이션 사전. RAG는 사내망 인프라가 안 받쳐주고 갱신·디버깅이 불투명해서 (b) 사전 + 자동 보강 조합으로 갔다. 구체적으로:

- **Layer 1 업무 사전(YAML)**: 운영자가 주요 테이블·컬럼·저장 함수에 한국어 업무명을 매핑.
- **Layer 2 물리 메타데이터**: `ALL_TAB_COLUMNS`/`ALL_INDEXES`/`ALL_CONSTRAINTS`에서 자동 수집.
- **Layer 3 공통코드**: `TB_COMMON_CODE`에서 코드그룹-값-명을 끌어와 LLM에게 주입.
- **Layer 4 지식 힌트(YAML)**: 운영 중 발견한 함정(대용량 테이블 WHERE 강제, 조인 키, 금지 패턴, 검증된 샘플 쿼리)을 점진적으로 추가.

운영자가 관리해야 하는 건 1·4번뿐이고 2·3은 서버 기동 때 자동 갱신.

### 에이전트 호스팅 + 서버리스 Python

별도 백엔드 서버를 세우지 않고 에이전트 플랫폼이 Python 앱을 직접 실행하게 했다. 사내 환경에서 항시 떠있는 서비스를 추가로 운영하기 부담스러웠고, 플랫폼이 어차피 호스팅을 해주니 코드만 등록하면 된다.

## 스택과 구조

- **Python 3.11+**: 에이전트 런타임이 받는 언어, 데이터 처리·테스트 생태계가 두텁다.
- **oracledb**: cx_Oracle 후속, thin 모드라 사내 PC에 Oracle Instant Client를 깔지 않아도 된다. 이관 부담을 크게 줄였다.
- **PyYAML**: 업무 사전·지식 힌트를 사람이 손으로 편집하는 파일 포맷.
- **sqlparse**: SELECT 검증·다중 구문 탐지·LIKE 앞 와일드카드 검출에 사용.
- **pytest + pytest-cov**: 검증 규칙·쿼리 실행기·도구 핸들러 단위·E2E 테스트.
- **선택 의존성**: FastAPI/uvicorn(관리자 페이지), click(CLI). 첫 단계에선 안 깔아도 동작.

소스 구조는 `core/metadata`(수집·카탈로그·사전·지식), `core/query`(검증·실행·포매터), `core/tools`(MCP 도구 세 개), `db`(Oracle 커넥션), `config`(환경변수·안전규칙·스키마 설정), `logging`(쿼리 실행 로그)으로 나눴다. 진입점은 `src/querycreator/app.py`의 `QueryCreatorApp` 한 개 클래스.

## 일화·디테일

- 안전 규칙은 7개로 떨어졌다. SELECT 전용, SELECT * 금지, 앞자리 와일드카드 LIKE 금지, 대용량 테이블 WHERE 강제, 30초 타임아웃, 1000행 상한, 다중 구문 금지. 1000행 초과는 잘라내고 응답에 `truncated: true`를 박는다.
- 대용량 테이블 판정은 `ALL_TABLES.NUM_ROWS`를 그대로 쓰지 않고, 운영자가 `data/knowledge/<SCHEMA>_hints.yaml`의 `large_tables`에 명시한 것만 강제한다. NUM_ROWS는 통계가 안 돌면 0으로 잡혀서 오탐이 나기 쉽다.
- 시스템 프롬프트(`prompts/system_prompt.md`)에 "제조 공정 데이터" 도메인을 박아놓았다. 공정 진행량 합산 같은 정형 계산은 저장 함수(`F_SUM_PROGRESS` 등)로 넘기게 유도한다.
- 첫 도메인이 주문(`TB_ORDER`) → 공정 진행 → 공통코드 번역 순서였다. "완료(03)" 식으로 코드값과 한국어 명칭을 같이 보여주는 응답 가이드가 여기서 굳어졌다.
- 테스트는 17개 파일, executor·validator·formatter·dictionary·knowledge·tools·E2E까지 분리해서 깔았다. coverage가 작업 중에는 잘 보였다.

## 회고

설계 단계에서 메타데이터를 4계층으로 쪼갠 게 가장 잘된 결정이다. 운영자가 만질 영역(사전·지식 힌트)과 자동화 영역(물리·공통코드)이 깔끔히 분리돼서, 새 스키마를 붙일 때 체크리스트가 단순해진다. 스키마별 수평 전개가 가능한 구조라, 다른 업무 스키마를 차례로 얹을 여지가 열려 있다.

도구 인터페이스를 MCP에 맞춘 것도 두고두고 이득. 사내 LLM 게이트웨이에서 직접 호출이든 에이전트 플랫폼 경유든 같은 서버 하나로 받는다.

해결 안 한 건 권한 모델이다. 지금은 SELECT 전용 DB 계정 한 개로 모든 사용자의 모든 조회를 받는다. 부서별 데이터 격리나 사용자 단위 감사 로그는 미정. 일단 쿼리 실행 로그(`logging/query_log.py`)에 사용자 질문 원문·생성 SQL·실행 시간을 남겨두고는 있는데, 사용자 식별자는 에이전트 쪽이 줘야 매핑된다.

또 하나는 LLM 모델 변동성. 사내 LLM 게이트웨이가 소넷/GPT-5.4/Gemini 3.1을 모드 전환으로 쓰는데, 같은 질문에 모델별로 다른 SQL을 짜낸다. 검증 레이어가 잡아주긴 하지만 응답 품질 편차는 별도로 회고할 거리.
